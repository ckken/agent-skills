# 外部 Agent 框架生命周期

适用于 Superpowers、Ponytail、Compound Engineering 等以插件、Skills、hook、模板、命令或项目目录提供工作流的外部框架。目标是降低提示词负担和流程冲突，同时避免把“精简”扩大成未经授权的全局删除。

## 1. 先识别实际状态

对每个框架记录：

| 项目 | 需要确认 |
|---|---|
| 安装形态 | plugin、marketplace、原生 Skill、软链接、clone、安装脚本或手工复制 |
| 加载形态 | 启用状态、session hook、项目 hook、自动触发 Skill、命令或显式调用 |
| 作用域 | 当前项目、用户级、系统级或某个特定宿主 |
| 载体 | `AGENTS.md`、`.codex/`、`.agents/`、`.superpowers/`、`docs/`、manifest、缓存和生成物 |
| 所有者 | 用户可管理、项目共享、外部托管或不可控系统层 |
| 替代关系 | 保留、已有替代、需要迁移或只是历史记录 |

不要只根据配置文件判断运行时状态。能查询宿主的插件、marketplace、hook 或 Skill 发现结果时，优先验证实时状态。

## 2. 选择目标生命周期

- **保留**：框架仍提供独特价值，且没有明显冲突或不必要常驻负担。
- **按需化**：保留能力，但移出稳定提示前缀，只在任务命中时加载。
- **停用**：关闭加载或触发，保留安装、数据和回滚能力。
- **迁移**：先将必要规则、计划、状态和验证合同转移到目标工作流，再处理旧框架。
- **卸载**：移除当前宿主中的可执行安装与加载入口。
- **深度清理**：在卸载基础上清理缓存、clone、隐藏目录、命令、术语和流程残留。

默认优先级是按需化或停用。只有用户明确授权相应动作后，才能迁移、卸载或深度清理。

## 3. 制定迁移与清理清单

按安装形态分别列出动作，不假设一条卸载命令覆盖全部状态：

1. 运行时：plugin、marketplace、hook、Skill 链接、命令和启动注入。
2. 用户目录：clone、缓存、插件数据、临时 marketplace 和旧 bootstrap。
3. 项目目录：Agent 配置、隐藏目录、计划、模板、生成物和工具说明。
4. 语义层：仍要求使用旧框架的规则、任务模板、命令示例和品牌术语。
5. 多宿主：Codex、Claude、Cursor 或其他宿主分别处理，不跨宿主推断授权。

对于 Superpowers，可能同时存在 Codex plugin、原生 Skill 软链接、clone 或旧 bootstrap 安装。应先识别当前形态，再选择宿主提供的卸载方式；不要把示例路径或历史命令当作当前事实。

## 4. 保护应保留内容

以下内容不能因包含框架名称而机械删除：

- 仍有效的架构决策、需求、验收标准和故障证据；
- 需要迁移到新工作流的任务状态；
- 法务、审计或发布所需历史记录；
- 被其他项目、宿主或用户共享的 marketplace、clone 和缓存；
- 用户未授权处理的用户级或系统级安装。

如果历史文档同时包含业务事实和旧框架术语，优先中性化或迁移业务事实，而不是整文件删除。

## 5. 验证完成状态

根据目标状态执行最小充分验证：

- **按需化**：稳定提示前缀不再常驻，相关任务仍能发现并使用能力。
- **停用**：新会话不再加载，安装和回滚入口仍存在。
- **迁移**：替代工作流覆盖原有必要行为，核心任务评测不下降。
- **卸载**：目标宿主不再列出或发现该框架，加载入口和链接已移除。
- **深度清理**：卸载成立，且授权范围内的缓存、clone、隐藏文件和语义残留已处理。

项目残留检查应包含隐藏文件，例如：

```bash
rg --hidden -n -i '<framework-name|commands|distinctive-terms>' <project-root>
```

搜索命中是人工复核线索，不等于必须删除。完成报告要分别说明运行时状态、文件状态、未处理宿主、保留历史资料和回滚方式。
