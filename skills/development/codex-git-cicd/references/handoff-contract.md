# Handoff 数据合同

handoff 的目标是让新的 Agent 在不依赖旧会话推理的情况下，从稳定检查点继续执行。事实、判断和待办必须分开。

## 必填区块

1. **目标与完成标准**：用户要达成什么，哪些远端结果才算完成。
2. **授权与边界**：允许修改、提交、推送、发布的范围；必须保留的内容。
3. **仓库快照**：仓库根、分支、HEAD、upstream、ahead/behind、detached/conflict 状态。
4. **改动分组**：staged、unstaged、untracked、conflicted；标出目标与非目标。
5. **验证证据**：命令、状态、关键结果和执行时点。
6. **远端状态**：push、PR、required checks、merge、tag、release、deploy 的真实状态。
7. **关键决策**：已经选择的方案、原因和被否决的替代方案。
8. **阻塞与风险**：外部依赖、权限、凭据、策略或未验证假设。
9. **恢复入口**：工作目录、需要读取的规则、第一组只读命令。
10. **下一条原子动作**：只写一条最先执行的动作，并给出完成条件。

## 状态词

- `PASS`：命令或门禁已通过。
- `FAIL`：已执行且失败，必须附根因或当前证据。
- `SKIP`：有意跳过，必须说明授权或风险依据。
- `BLOCKED`：因外部状态无法继续，必须写解除条件。
- `UNKNOWN`：尚未查询，禁止推断为成功。

## 恢复规则

- 新 Agent 先读取仓库指令和 handoff，再重新运行只读体检。
- 对容易漂移的状态重新查询：远端 HEAD、CI、PR、审批、merge 和部署。
- 不重复已经由 commit 哈希和日志证明完成的本地工作。
- handoff 与实时状态冲突时，以实时状态为准，并更新交接包。
- 任何包含 Token、私有 URL、个人数据或密钥值的内容都不得写入 handoff。

## 建议刷新策略

稳定检查点覆盖更新同一文件；需要审计轨迹时另存带时间戳副本。交接文件默认不提交，除非仓库已把它定义为项目工作流资产。
