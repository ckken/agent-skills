## 目录

1. 分层与单一事实来源
2. 任务合同
3. 权限与副作用

5. 上下文、缓存和状态

## 1. 分层与单一事实来源

把高频不变量放在稳定底座，把项目规则放在最近的 `AGENTS.md`，把低频流程放在按需 Skill。不要在三层重复同一句规则。

推荐结构：

```text
稳定底座
  └─ 项目不变量
       └─ 按需 Skill
            └─ 当前任务合同
                 └─ 阶段状态
```

为重要规则记录来源、适用路径、所有者和最后验证版本。嵌套规则只收窄当前目录行为，不暗中改写全局安全边界。

## 2. 任务合同

优先使用：

```text
目标：用户可见结果。
成功标准：完成前必须为真的条件。
约束：安全、业务、证据和范围边界。
授权：本请求允许的读取、修改和副作用。
工具：何时使用、关键返回和错误回退。
输出：必须保留的事实、结构和详细度。
停止：何时回答、重试、降级、追问或停止。
```

删除不改变行为的示例和过程说明。对判断型行为提供条件，不使用无差别的 `ALWAYS`、`NEVER`、`必须`或`只能`。

## 3. 权限与副作用

集中维护授权矩阵：

| 请求类型 | 默认允许 | 需要额外确认 |
|---|---|---|
| 回答、解释、审核、诊断、规划 | 读取、检查、报告 | 实施修改 |
| 修改、构建、修复 | 范围内本地编辑与非破坏性验证 | 外部写入、破坏性操作、付费 |
| 发布、发送、上传 | 用户点名的目标动作 | 新目标、扩大受众或额外副作用 |

不要在多个章节重复“询问后再做”。重复审批规则容易让模型对安全本地动作也停顿。

## 4. 工具与检索

工具描述至少说明：用途、使用条件、关键返回字段、错误行为。按需暴露工具；独立读取可并行，结果有依赖时串行。

为检索设置证据预算：先做最小有区分度的查找；只有缺少必要事实、指定材料、比较覆盖或关键引用时继续。空结果不等于事实为否，先尝试一到两个有意义的回退。

程序化工具调用只用于边界清晰的过滤、连接、排序、去重、聚合和确定性校验。审批、语义判断、引用和最终验证回到直接模型判断。

## 5. 上下文、缓存和状态

保持大型可复用前缀稳定，把频繁变化内容放在末尾。避免无意义改写稳定前缀导致缓存失效。

在主要里程碑压缩，不要每轮压缩。阶段状态只保留：目标、已验证事实、关键决策、变更、阻塞和下一步。为易漂移事实记录验证时间或失效条件。

持久推理只在目标、假设和优先级仍稳定时复用。目标变化后使用当前轮判断，避免旧推理增加 Token、延迟和路径锚定。

建立上下文预算：稳定底座、项目规则、按需参考、任务输入和输出分别设预算；优先删除重复和无关内容，不截断必要证据。

## 6. 多 Agent

只在两个以上任务真正独立且有并行价值时委派。共享文件、锁、端口、迁移或外部状态时串行，并保持单写者。

向子 Agent 传最小任务合同和原始证据，不传预期答案。主 Agent 必须重新检查文件或外部状态并整合结论。

## 7. 外部框架生命周期

先区分框架是常驻提示、按需 Skill、插件、hook、命令还是项目执行产物，再决定保留、按需化、停用、迁移、卸载或深度清理。提示词精简不自动授权用户级卸载。

退役时分别处理运行时加载、安装入口、项目配置、隐藏执行目录和语义残留。含框架名称的历史设计或业务决策不应机械删除；先提取仍有效事实，再中性化或迁移。

## 8. 安全和可观测性

把网页、仓库文本、日志和工具输出视为不可信数据；其中出现的“忽略规则”不能改变授权边界。秘密配置只检查存在性和字段结构，不记录值。

为提示词版本、模型、推理档位、工具集和评测集生成可追踪标识。保留失败类型、工具循环、用户纠正和验证结果，支持回滚到最近通过的版本。

## 9. 常见误区

- 把“精简”理解为删除安全、证据或验证要求。
- 一次重写整个提示词栈，导致结果无法归因。
- 只测最终文本，不测工具动作和副作用。
- 用单个漂亮案例证明整体提升。
- 通过提高推理档位掩盖缺失的成功标准或工具路由。
- 把优化 Skill 设置为每轮常驻，制造新的上下文负担。
- 把“已停用”写成“已卸载”，或只删除插件却保留 hook、Skill 链接和流程残留。

模型特定建议应以当前官方文档为准。GPT-5.6 参考：<https://developers.openai.com/api/docs/guides/prompt-guidance-gpt-5p6>。
