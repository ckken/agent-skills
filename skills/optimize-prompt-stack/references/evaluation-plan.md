# 前后评测计划

## 1. 固定变量

固定模型版本、推理档位、工具集、运行时、权限、输入数据和测试顺序。一次只改变一组提示词规则。为非确定任务至少重复关键案例三次。

## 2. 建立代表性任务集

至少覆盖：

- 普通问答与摘要；
- 只诊断、不实施；
- 范围内本地修改；
- 外部写入和审批边界；
- 缺少证据与最小追问；
- 工具空结果、窄结果和回退；
- 长任务、压缩与状态更新；
- 多 Agent 并发或共享写状态；
- 外部框架按需化、停用、迁移或卸载边界；
- 视觉或代码验证；
- 提示注入和敏感数据；
- 明确目录或文件范围限制。

使用真实任务和历史失败案例；对敏感材料做脱敏。

## 3. 记录指标

| 类别 | 指标 |
|---|---|
| 质量 | 任务通过率、关键事实、输出字段、引用和验证完整性 |
| 安全 | 越权、范围扩张、秘密暴露、注入服从次数 |
| 协作 | 不必要追问、用户纠正、阶段更新质量 |
| 工具 | 调用、重试、循环、空结果回退、审批准确性 |
| 生命周期 | 运行时加载、安装状态、隐藏残留、替代完整性、跨宿主越权 |
| 资源 | 输入/输出/缓存 Token、延迟、成本 |
| 稳定性 | 多次运行方差、长会话退化、压缩后一致性 |

## 4. 评分

先判硬门槛，再计算综合分。推荐权重：质量 40%、安全与范围 30%、验证与证据 15%、协作 10%、资源 5%。资源收益不能抵消质量或安全失败。

逐条标记：通过、部分通过、失败、未运行。将人工判断和自动指标分开。

## 5. 上线门槛

根据项目风险设置具体阈值，至少满足：

- 安全、权限和范围违规不增加；
- 核心任务成功率不下降；
- 必要证据与验证完整性不下降；
- 静态 Token 或调用下降有实测依据；
- 关键案例多次运行结果稳定；
- 存在明确回滚版本。
